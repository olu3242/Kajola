// What Kajola Generates

Every deliverable
your team needs

Kajola doesn't produce summaries or outlines. It produces the real thing — production SQL with RLS, typed API specs, and deployment steps your engineering team can execute immediately.

01

Product Requirements Document

Vision, personas, P0/P1/P2 prioritization, KPIs, and explicit out-of-scope decisions.

02

System Architecture Diagram

ASCII diagram of all layers: client apps, edge functions, DB, queue, storage, external integrations.

03

Full SQL Schema

Every table with enums, constraints, indexes, foreign keys, RLS policies, and trigger functions.

04

API Endpoint Definitions

Method, path, auth requirement, request/response shape, and error cases per endpoint.

05

Frontend Component Structure

Full directory tree for Next.js dashboard and Expo mobile app with component descriptions.

06

Monorepo Folder Layout

Turborepo structure: apps, packages, supabase functions, shared UI and API layers.

07

Automation Engine Design

Event table, rule schema, idempotency pattern, retry logic, and cron-based failure recovery.

08

Deployment Plan

Step-by-step Supabase setup, all environment variables, and CI/CD pipeline specification.

09

Monetization Strategy

Transaction fees, subscription tiers, featured listings — with specific rates, not generic models.

10

Scaling Strategy

Concrete triggers: what changes at 10k, 100k, and 1M users.

11

Product Roadmap

Quarterly feature plan with business impact framing, not just a feature list.

// Why Kajola

Built for the
real world

🔒

Tenant Isolation by Default

Every table gets tenant_id. Every query is wrapped in an RLS policy using the current_user_tenant_id() helper function. Super admin bypass is included. Cross-tenant data leaks are architecturally impossible.

🌍

Africa-First Patterns

SMS over email. Paystack + Flutterwave + M-Pesa payment abstraction layer. Offline action queue. Low-bandwidth mode. Android-first mobile UI guidance baked in.

⚡

Event-Driven Automation

Full automation engine: system_events table, rule engine, idempotent execution, retry logic with exponential backoff. Booking → SMS → Review — all wired.

🔄

No Placeholders Allowed

The skill enforces a zero-placeholder quality standard. Every SQL index is named. Every API endpoint has error cases. Every env var is listed with its purpose.

📐

Domain-Adaptive

Works for booking platforms, equipment rentals, fintech, edtech, and more. The architecture patterns flex to the domain — the quality standard never does.

// Example Output

Real SQL,
not pseudocode

Kajola generates the actual CREATE TABLE statements, RLS policies, conflict-prevention triggers, and index definitions your database needs — not annotated skeletons for you to complete.

0001_initial.sql
-- Booking conflict prevention trigger CREATE OR REPLACE FUNCTION prevent_booking_overlap() RETURNS TRIGGER AS $$ BEGIN IF EXISTS ( SELECT 1 FROM bookings WHERE artisan_id = NEW.artisan_id AND status NOT IN ('cancelled','no_show') AND scheduled_start < NEW.scheduled_end AND scheduled_end > NEW.scheduled_start ) THEN RAISE EXCEPTION 'Booking conflict detected'; END IF; RETURN NEW; END; $$ LANGUAGE plpgsql; -- RLS: tenant isolation CREATE POLICY "tenant_isolation" ON bookings FOR ALL USING (tenant_id = current_user_tenant_id()); CREATE POLICY "super_admin_bypass" ON bookings FOR ALL USING (has_role('super_admin'));
