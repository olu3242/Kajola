## Section 1 — Product Requirements Document

### 1.1 Vision

Nigeria's construction sector generates over ₦15 trillion in annual activity, yet equipment access remains the industry's biggest bottleneck. Small and mid-size contractors spend weeks sourcing a generator or excavator through informal calls, with no pricing transparency, no verified equipment condition, and no protection against disputes. Equipment owners — individuals and fleet companies alike — have idle assets earning nothing between jobs. ToolHire Pro bridges this gap: owners list equipment with verified specs and availability; contractors discover, book, and pay securely; delivery is coordinated end-to-end on platform. In three years, ToolHire Pro is the default equipment layer for Nigerian construction — the first call every site manager makes when they need kit, with 10,000 active listings and ₦500M+ in monthly GMV.

### 2.3 Data Flow Narratives

#### Flow 1: User Registration (Phone OTP)
1. User enters phone number (e.g. `+2348031234567`) in app
2. App calls Edge Function `POST /auth/send-otp` with `{ "phone": "+2348031234567" }`
3. Edge Function checks rate limit: max 3 OTPs per phone per 10 minutes (queries `phone_otps` table)
4. Generates 6-digit OTP, bcrypt-hashes it, inserts into `phone_otps` with 10-minute expiry
5. Calls Termii `POST /api/sms/otp/send` with plaintext OTP and sender ID `ToolHire`
6. Termii delivers SMS: "Your ToolHire Pro code is 839204. Valid 10 minutes. Do not share."
7. User enters code; app calls `POST /auth/verify-otp`
8. Edge Function finds unexpired row, compares bcrypt hash, marks `verified_at`
9. Calls `supabase.auth.admin.createUser` (new) or signs in (returning); returns JWT pair
10. Edge Function creates `users` row and empty `owner_profiles` or `renter_profiles` row
11. App stores tokens in Expo SecureStore; navigates to role-selection onboarding

#### Flow 2: Equipment Booking + Payment (including deposit)
1. Renter searches for "20KVA generator Lagos" → selects listing from Chukwuemeka's fleet
2. App calls `GET /equipment/{id}/availability?from=2026-05-01&to=2026-05-07` → available
3. Renter selects 7-day rental (₦35,000) + delivery (₦5,000) + deposit (₦50,000) = ₦90,000 total
4. App calls `POST /rentals/create` with equipment ID, dates, delivery address
5. Edge Function validates availability (DB exclusion constraint handles conflicts atomically)
6. Creates `rentals` row (status: `pending`), creates two `transactions` rows: one for rental + delivery, one for deposit
7. Calls Paystack `POST /transaction/initialize` for combined amount ₦90,000
8. Returns `checkout_url` to app; app opens Paystack WebView
9. Renter completes card or bank transfer payment
10. Paystack fires `charge.success` webhook to `POST /webhooks/paystack`
11. Webhook handler: verifies HMAC, checks idempotency, logs to `webhook_logs`
12. Enqueues `payment.confirmed` automation job
13. Job: updates `rentals.status = 'confirmed'`, marks both transactions `success`
14. Supabase Realtime broadcasts rental status change to owner and renter apps
15. Termii SMS to owner: "New rental: 20KVA Generator, 1–7 May. ₦35,000 + ₦5,000 delivery. Renter paid ₦90,000 incl. deposit."

#### Flow 3: Condition Check-out and Check-in
**Check-out (before delivery):**
1. Owner taps "Start Rental" in app → rental status: `delivery_in_progress`
2. Owner uploads up to 10 condition photos via `POST /rentals/{id}/checkout-photos`
3. Photos stored in `condition-photos/{rental_id}/checkout/` in Supabase Storage (private bucket)
4. Edge Function creates `condition_records` row (type: `checkout`), links photo paths
5. Renter receives push: "Your equipment is on its way! Review check-out photos." — opens signed URLs
6. Renter confirms receipt → status: `active`

**Check-in (on return):**
1. Owner uploads return photos via `POST /rentals/{id}/checkin-photos`
2. Edge Function creates `condition_records` row (type: `checkin`)
3. Owner marks return as clean or damaged with optional damage description
4. If clean: automation job `deposit.release` fires — full ₦50,000 returned to renter's card via Paystack refund
5. If damaged: owner specifies deduction amount (e.g. ₦15,000); renter is notified and has 24h to accept or dispute
6. If accepted: ₦35,000 refunded to renter, ₦15,000 credited to owner's wallet
7. If disputed: status → `deposit_disputed`; admin reviews both photo sets and adjudicates

#### Flow 4: Payout to Equipment Owner
1. On `rentals.status = 'completed'`: automation job `payout.rental` fires
2. Calculates: rental fee ₦35,000 + delivery ₦5,000 = ₦40,000 gross
3. Platform fee 10% = ₦4,000; Paystack processing ~1.5% = ₦600
4. Net payout to owner: ₦35,400
5. Calls Paystack `POST /transfer` to owner's recipient code (`owner_profiles.paystack_recipient_code`)
6. Credits `wallets.balance_kobo` for owner, creates `wallet_transactions` row
7. Termii SMS to owner: "₦35,400 sent to your Access Bank account ****7823. Rental: 20KVA Gen, 1–7 May. Ref: THP-TRF-00187"
8. Deposit payout handled separately via `deposit.release` job (above)

CREATE OR REPLACE FUNCTION update_equipment_search_vector()
RETURNS TRIGGER AS $$
BEGIN
  NEW.search_vector := to_tsvector('english',
    coalesce(NEW.title, '') || ' ' ||
    coalesce(NEW.description, '') || ' ' ||
    coalesce(NEW.make, '') || ' ' ||
    coalesce(NEW.model, '') || ' ' ||
    coalesce(NEW.location_text, '') || ' ' ||
    coalesce(NEW.category::text, '')
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

  BEFORE INSERT OR UPDATE ON equipment

  id                    uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  tenant_id             uuid NOT NULL REFERENCES tenants(id),
  renter_id             uuid NOT NULL REFERENCES users(id),
  owner_id              uuid NOT NULL REFERENCES owner_profiles(id),
  equipment_id          uuid NOT NULL REFERENCES equipment(id),
  status                rental_status NOT NULL DEFAULT 'pending',
  starts_on             date NOT NULL,
  ends_on               date NOT NULL CHECK (ends_on >= starts_on),
  rental_days           smallint NOT NULL GENERATED ALWAYS AS (ends_on - starts_on + 1) STORED,
  delivery_address_text text,
  delivery_location     geography(POINT, 4326),
  rental_fee_kobo       bigint NOT NULL CHECK (rental_fee_kobo > 0),
  delivery_fee_kobo     bigint NOT NULL DEFAULT 0 CHECK (delivery_fee_kobo >= 0),
  deposit_kobo          bigint NOT NULL DEFAULT 0 CHECK (deposit_kobo >= 0),
  platform_fee_kobo     bigint NOT NULL DEFAULT 0 CHECK (platform_fee_kobo >= 0),
  owner_payout_kobo     bigint NOT NULL CHECK (owner_payout_kobo > 0),
  deposit_status        deposit_status NOT NULL DEFAULT 'held',
  deposit_deduction_kobo bigint NOT NULL DEFAULT 0 CHECK (deposit_deduction_kobo >= 0),
  deposit_deduction_reason text,
  renter_note           text,
  cancellation_reason   text,
  cancelled_by          uuid REFERENCES users(id),
  confirmed_at          timestamptz,
  delivered_at          timestamptz,
  returned_at           timestamptz,
  completed_at          timestamptz,
  -- Prevent double-booking at DB level
  CONSTRAINT no_double_rental EXCLUDE USING gist (
    equipment_id WITH =,
    daterange(starts_on, ends_on, '[]') WITH &&
  ) WHERE (status NOT IN ('cancelled')),
  created_at            timestamptz NOT NULL DEFAULT now(),
  updated_at            timestamptz NOT NULL DEFAULT now()
);

CREATE TRIGGER set_rentals_updated_at
  BEFORE UPDATE ON rentals FOR EACH ROW EXECUTE FUNCTION update_updated_at();

CREATE INDEX rentals_tenant_idx ON rentals (tenant_id, created_at DESC);
CREATE INDEX rentals_renter_idx ON rentals (renter_id, created_at DESC);
CREATE INDEX rentals_owner_idx ON rentals (owner_id, starts_on);
CREATE INDEX rentals_equipment_idx ON rentals (equipment_id, starts_on, ends_on);
CREATE INDEX rentals_status_idx ON rentals (status, created_at DESC) WHERE status IN ('pending','confirmed','active','deposit_disputed');

-- Auto-update equipment and owner average ratings
CREATE OR REPLACE FUNCTION refresh_equipment_rating()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.reviewee_type = 'equipment' THEN
    UPDATE equipment SET
      average_rating = (SELECT ROUND(AVG(rating)::numeric, 2) FROM reviews WHERE reviewee_type = 'equipment' AND reviewee_id = NEW.reviewee_id AND is_visible = true),
      total_reviews  = (SELECT COUNT(*) FROM reviews WHERE reviewee_type = 'equipment' AND reviewee_id = NEW.reviewee_id AND is_visible = true)
    WHERE id = NEW.reviewee_id;
  ELSIF NEW.reviewee_type = 'owner' THEN
    UPDATE owner_profiles SET
      average_rating = (SELECT ROUND(AVG(rating)::numeric, 2) FROM reviews WHERE reviewee_type = 'owner' AND reviewee_id = NEW.reviewee_id AND is_visible = true),
      total_reviews  = (SELECT COUNT(*) FROM reviews WHERE reviewee_type = 'owner' AND reviewee_id = NEW.reviewee_id AND is_visible = true)
    WHERE id = NEW.reviewee_id;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER update_ratings_after_review
  AFTER INSERT OR UPDATE ON reviews
  FOR EACH ROW EXECUTE FUNCTION refresh_equipment_rating();
